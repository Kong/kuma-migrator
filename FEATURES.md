## 1. CLI Commands

### `kuma-migrator extract`

Extracts Kuma policy resources from a running control plane and writes one YAML file per resource.

| Flag | Short | Required | Description |
|---|---|---|---|
| `--kube-context` | | mutually exclusive | Kubernetes context; uses `kubectl` |
| `--kumactl-context` | | mutually exclusive | kumactl context from `~/.kumactl/config` (or `$KUMACTL_CONFIG`) |
| `--output-dir` | `-o` | yes | Directory to write extracted YAML files |

**Kubernetes mode** — lists all `kuma.io/v1alpha1` CRDs and fetches every instance with `kubectl get <kind> -A -o json`.

**Kumactl mode** — resolves the context from the kumactl config file, calls `GET <cpURL>/_resources` to discover all writable resource types (`readOnly: false`), lists all Mesh names, then calls `kumactl get <type> [--mesh <mesh>] -o yaml` for each type × mesh combination.

---

### `kuma-migrator plan`

Previews all migration changes in dry-run mode. No output YAML files are written.

| Flag | Short | Required | Description |
|---|---|---|---|
| `--input-dir` | `-i` | yes | Directory containing source policy YAML files |
| `--output-dir` | `-o` | yes | Directory to write `migration-plan.md` |

---

### `kuma-migrator migrate`

Transforms source YAML files and writes migrated output. Produces a full Markdown report.

| Flag | Short | Required | Description |
|---|---|---|---|
| `--input-dir` | `-i` | yes | Directory containing source policy YAML files |
| `--output-dir` | `-o` | yes | Directory to write migrated YAML files and `migration-report.md` |

---

### `kuma-migrator version`

Prints the tool version (`kuma-migrator <version>`).

## 2. Extract — CP Mode Detection

The tool detects the control plane mode at extract time and adjusts filtering accordingly.

### Kubernetes path

Reads the `KUMA_MODE` and `KUMA_MULTIZONE_ZONE_NAME` environment variables from the CP Deployment.
Tries both `kuma-system/kuma-control-plane` and `kong-mesh-system/kong-mesh-control-plane`.
Falls back to `""` (unknown) if neither Deployment is found.

### Kumactl path

Calls `GET <cpURL>/config` and reads `mode` and `multizone.zone.name` from the JSON response.
Falls back to `""` (unknown) on any HTTP or parse error.

### Modes

| Mode | Behaviour |
|---|---|
| `global` | All resources extracted; zone-only kinds skipped |
| `standalone` | All resources extracted (single-CP deployment) |
| `zone` | Only `kuma.io/origin: zone` resources kept; see [zone filtering](#3-extract--resource-filtering) |
| `""` (unknown) | All resources extracted; warning printed |

### Output directory naming

| Mode | Output subdirectory |
|---|---|
| `global` | `<output-dir>/global/` |
| `standalone` | `<output-dir>/standalone/` |
| `zone` | `<output-dir>/zone-<zone-name>/` (e.g. `zone-eu-west`) |
| `""` | `<output-dir>/unknown/` |

## 3. Extract — Resource Filtering

### Insight kinds — always skipped

Any resource whose `kind` contains `"Insight"` (e.g. `ZoneInsight`, `DataplaneInsight`) is skipped unconditionally.

### Default skip list

The following kinds are skipped by default. The list is user-configurable via the [config file](#11-config-file).

`Dataplane`, `AccessRole`, `AccessRoleBinding`, `ZoneEgress`, `ZoneIngress`, `Zone`, `HostnameGenerator`, `Workload`

### MeshService auto-generated resources

`MeshService` resources with the label `kuma.io/env: kubernetes` are skipped. These are auto-generated by Kuma from Kubernetes `Service` objects and do not need to be migrated.

### Zone CP origin filter

On a Zone CP, the origin label determines whether a resource is kept:

| `kuma.io/origin` value | Action |
|---|---|
| `zone` | Kept |
| `global` | Skipped (synced from Global CP; extract from there instead) |
| absent — gateway-local kind | Kept (see below) |
| absent — any other kind | Skipped |

**Gateway-local kinds** may legitimately lack an origin label on a Zone CP when created directly there (not via the Global CP). These are kept when the origin label is absent but still skipped if `kuma.io/origin: global` is present:

`MeshGatewayInstance`, `MeshGatewayConfig`, `MeshGateway`, `MeshHTTPRoute`, `MeshTCPRoute`, `MeshGatewayRoute`

### Global CP zone-only filter

On a Global CP, the following kinds are always skipped — they are zone-local and never synced to the Global CP:

`MeshGatewayInstance`, `MeshGatewayConfig`

### Gateway resource placement

| Kind | Typical origin | Global CP extraction | Zone CP extraction |
|---|---|---|---|
| `MeshGateway`, `MeshHTTPRoute`, `MeshTCPRoute`, `MeshGatewayRoute` | Global CP | extracted normally | skipped (`kuma.io/origin: global`) |
| `MeshGateway`, `MeshHTTPRoute`, `MeshTCPRoute`, `MeshGatewayRoute` | Zone CP (less common) | not present | extracted (no origin label) |
| `MeshGatewayInstance`, `MeshGatewayConfig` | Zone CP | **always skipped** | extracted (may have no origin label) |

## 4. Extract — Output Structure

```
<output-dir>/
  global/
    resiliency/
    routing/
    zero-trust/
    observability/
    mesh/
  zone-<zone-name>/
    resiliency/
    routing/
    ...
```

### Migrate — all-zones routing

When migrating resources from the `global/` input folder, Gateway API output kinds (`Gateway`, `HTTPRoute`, `TCPRoute`, `GatewayClass`, `ReferenceGrant`) are written to `all-zones/` instead of `global/`. These are Kubernetes-native CRDs that must be applied to zone clusters, not to the Global CP.

```
<output-dir>/
  global/          ← Kuma policy CRDs; apply to Global CP
  all-zones/       ← Gateway API CRDs; apply to every Zone cluster
    routing/
  zone-<name>/     ← zone-origin Kuma policies; apply to that Zone cluster
    resiliency/
    ...
```

## 5. Migration Scenarios

### Scenario A — Legacy

**Trigger**: `sources`/`destinations` keys present, or a known legacy type name (`Timeout`, `CircuitBreaker`, `Retry`, `TrafficPermission`, `FaultInjection`, `RateLimit`, `HealthCheck`, `TrafficLog`, `TrafficTrace`, `ProxyTemplate`).

**Transformations**:
- Universal YAML → Kubernetes-style with `apiVersion`/`kind`/`metadata`/`spec`
- `sources` → `spec.targetRef`; `destinations` → `spec.to[]`
- `TrafficPermission` semantics are inverted: `destinations` → `spec.targetRef`, `sources` → `spec.from[]`
- Multiple specific sources are split into one output policy per source (named `<original>-0`, `<original>-1`, …)
- Wildcard source (`kuma.io/service: '*'`) → `spec.targetRef.kind: Mesh`

**Type mapping**:

| Old type | New kind |
|---|---|
| `Timeout` | `MeshTimeout` |
| `CircuitBreaker` | `MeshCircuitBreaker` |
| `Retry` | `MeshRetry` |
| `TrafficPermission` | `MeshTrafficPermission` |
| `FaultInjection` | `MeshFaultInjection` |
| `RateLimit` | `MeshRateLimit` |
| `HealthCheck` | `MeshHealthCheck` |
| `TrafficLog` | `MeshAccessLog` |
| `TrafficTrace` | `MeshTrace` |
| `ProxyTemplate` | `MeshProxyPatch` |
| `TrafficRoute` | **Error** — ambiguous (HTTP vs TCP); requires manual migration |

---

### Scenario B — Subset

**Trigger**: New `Mesh*` policy using `MeshSubset` with `kuma.io/service` or `k8s.kuma.io/service-name` tags in any `targetRef`.

**Transformations**:
- All `targetRef` entries with service-identity tags are rewritten
- `spec.targetRef` (top-level) → `kind: Dataplane` with `labels: {app: <name>}` (best-guess; warning emitted)
- `to[]`/`from[]` entries → `kind: MeshService` with `name`/`namespace` (or `labels` when namespaces differ)
- Wildcard tag (`*`) → `kind: Mesh`
- When `from[]` is also present, a Rules API pass is applied after the Subset rewrite (see Scenario D)

---

### Scenario C — Passthrough

**Trigger**: Already using the new API with no service-identity tags; or not a recognised Kuma policy.

**Transformation**: Document passed through unchanged. Still scanned by the post-pass deprecation checker.

---

### Scenario D — Rules

**Trigger**: New-style `Mesh*` policy with a `from[]` section (deprecated in Kuma 2.10+). Affected kinds: `MeshAccessLog`, `MeshCircuitBreaker`, `MeshRateLimit`, `MeshTimeout`, `MeshTls`.

**Transformation**:
- Each `from[i].default` → `rules[i].default`
- The source `targetRef` inside `from[]` is discarded (`rules[]` has no source discrimination)
- `to[]` is preserved unchanged

**Warning**: If `from[]` contained multiple entries with different source `targetRef` kinds, collapsing them into `rules[]` loses per-source configuration — a warning is emitted.

---

### Scenario — Mesh

**Trigger**: `kind: Mesh` with embedded observability or passthrough configuration, or `spec.meshServices.mode` not set to `Exclusive`.

**Transformations**:
- `spec.metrics` → standalone `MeshMetric` resource(s)
- `spec.tracing` → standalone `MeshTrace` resource(s)
- `spec.logging` → standalone `MeshAccessLog` resource(s)
- `spec.networking.outbound.passthrough` → standalone `MeshPassthrough` resource
- Patches the `Mesh` CR to set `spec.meshServices.mode: Exclusive`

---

### Scenario — ExternalService

**Trigger**: `kind: ExternalService` or `type: ExternalService`.

**Transformation**: Converts to `MeshExternalService`.

**Error cases**:
- No name
- No `networking.address`
- Unix socket address (`unix://` or `/`) — removed in Kuma 2.9; manual migration required
- Invalid `host:port` format

---

### Scenario — Gateway (GW)

Five sub-scenarios, all labelled `[MIGRATED GW]`:

| Input kind | Output |
|---|---|
| `MeshGateway` | `Gateway` (Gateway API `gateway.networking.k8s.io/v1`), `gatewayClassName: gateways.kuma.io/controller` |
| `MeshGatewayInstance` | `GatewayClass` + `MeshGatewayConfig` |
| `MeshHTTPRoute` | `HTTPRoute` (Gateway API) |
| `MeshTCPRoute` | `TCPRoute` (Gateway API experimental) |
| `MeshGatewayRoute` | `HTTPRoute` and/or `TCPRoute` depending on protocol |

**Gateway-specific behaviours**:
- Listener `hostname: "*"` is invalid in Gateway API — field is omitted and a warning is emitted
- `kuma.io/service`-encoded backend names (e.g. `backend_demo_svc_3001`) are parsed to extract `name`, `namespace`, and `port` for `backendRef`/`parentRef`
- GAMMA mesh routing `parentRef` with `kind: Service` always includes the `port` field

---

### Scenario — OPAPolicy

**Trigger**: `kind: OPAPolicy` (Kong Mesh legacy OPA integration).

**Transformation** (Kong Mesh 2.5+):
- `spec.conf.policies[].inlineString` → `spec.default.appendPolicies[].rego.inlineString`
- `spec.conf.agentConfig.inlineString` → `spec.default.agentConfig.inlineString`
- `spec.targetRef` preserved as-is

## 6. Deprecation Warnings & Auto-fixes

`ScanForDeprecations()` is called on every output document regardless of scenario.

| Field / Pattern | Version | Auto-fix | Action |
|---|---|---|---|
| `MeshMetric` `spec.default.sidecar.regex` | v2.7 | **Yes** | Rewritten to `sidecar.profiles.exclude[0]` with `type: Regex` |
| `MeshHealthCheck` `healthyPanicThreshold` | v2.10 | No | Move value to `MeshCircuitBreaker.spec.default.outlierDetection` |
| `MeshTrust` `spec.origin` | v2.13 | No | Remove field; now read-only at `status.origin` |
| `MeshTrafficPermission`/`MeshFaultInjection` `from[].targetRef.kind: MeshService` | v2.7 | No | Replace with `kind: Dataplane` + labels |
| `MeshTrafficPermission` `action: ALLOW`/`DENY`/`ALLOW_WITH_SHADOW_DENY` (uppercase) | KM 2.1 | No | Use `Allow` / `Deny` / `AllowWithShadowDeny` |
| `MeshLoadBalancingStrategy` `hashPolicies[].type: SourceIP` | v2.10 | No | Replace with `Connection` |
| `Dataplane` `transparentProxying.redirectPortInboundV6` | v2.9 | No | Remove field |
| Any `Mesh*` policy `spec.targetRef.kind: MeshSubset` without service-identity tags | v2.10 | No | Replace with `kind: Dataplane` + labels |

## 7. Workload Scanners

Both scanners run on every document regardless of scenario. Hits are collected and surfaced in the report's Action Items section.

### Environment variable scanner

Detects legacy `kuma.io/service`-encoded addresses in workload container env vars.

**Supported workload kinds**: `Deployment`, `StatefulSet`, `DaemonSet`, `Job`, `CronJob`, `Pod`, `ReplicaSet`

**Pattern matched**: `<service-name>_<namespace>_svc_<port>[:<explicit-port>]`

**Per hit, reports**:
- Workload kind, name, namespace
- Container name and env var name
- Old token (e.g. `backend_demo_svc_3000`)
- New Kubernetes address: `<service-name>.<namespace>.svc.cluster.local[:port]`
- New mesh address: `<service-name>.<namespace>.svc.<zone>.mesh.local[:port]`

### Annotation scanner

Detects deprecated boolean annotation values on any Kubernetes resource.

**Deprecated values**: `"yes"` and `"no"` (deprecated in Kuma 2.9; use `"true"` / `"false"`)

**Prefix matched**: `kuma.io/` and `k8s.kuma.io/`

**Per hit, reports**: Kind, name, namespace, annotation key, old value, suggested new value.

## 8. RFC Name Validation

Applied to every input and generated resource name.

| Standard | Applied to | Rules |
|---|---|---|
| RFC 1035 (strict) | `Mesh`, `MeshService`, `MeshExternalService`, `MeshMultizoneService` | Must start with a letter; alphanumeric + hyphens only; max 63 chars |
| RFC 1123 (relaxed) | All other kinds | May start with digit; alphanumeric + hyphens; must not start or end with hyphen; max 63 chars |

Violations produce a warning; the resource is still written.

## 9. Resource Subfolder Mapping

Output files are organised into subfolders by kind.

| Subfolder | Kinds |
|---|---|
| `resiliency/` | `MeshCircuitBreaker`, `MeshRetry`, `MeshTimeout`, `MeshHealthCheck`, `MeshRateLimit`, `MeshFaultInjection` (and legacy equivalents) |
| `routing/` | `Gateway`, `GatewayClass`, `HTTPRoute`, `TCPRoute`, `MeshGatewayConfig`, `MeshGateway`, `MeshGatewayInstance`, `MeshGatewayRoute`, `MeshHTTPRoute`, `MeshTCPRoute`, `MeshLoadBalancingStrategy`, `MeshPassthrough`, `MeshExternalService`, `ExternalService`, `MeshService`, `MeshMultiZoneService`, `VirtualOutbound`, `TrafficRoute` |
| `zero-trust/` | `MeshTrafficPermission`, `MeshTLS`, `MeshIdentity`, `MeshTrust`, `MeshOPA`, `TrafficPermission`, `OPAPolicy`, `Secret`, `GlobalSecret` |
| `mesh/` | `Mesh`, `MeshProxyPatch`, `ProxyTemplate` |
| `observability/` | `MeshMetric`, `MeshTrace`, `MeshAccessLog`, `TrafficLog`, `TrafficTrace` |
| `other/` | All unrecognised kinds |

## 10. Report Format

Both `migration-plan.md` and `migration-report.md` share the same structure; plan mode uses simplified versions of some sections.

### Sections

| Section | Plan | Migrate | Notes |
|---|---|---|---|
| Header | ✓ | ✓ | Timestamp, input/output dirs; plan mode shows "Dry run" notice |
| Summary table | ✓ | ✓ | Total files, migrated, already migrated, skipped, errors |
| Migrated Files | ✓ | ✓ | Grouped by `<cpModeDir>/<subfolder>`; per-file warnings, workload hits, annotation hits |
| Already Migrated | ✓ | ✓ | Files passed through unchanged |
| Skipped | ✓ | ✓ | Files with no recognised Kuma policy documents |
| Action Items | ✓ | ✓ | Errors requiring manual intervention; workload address mapping table; deprecated annotation table |
| Apply Checklist | ✓ (short) | ✓ (full) | See below |
| Original Resources to Delete | ✓ (preview) | ✓ (full) | Resources whose kind changed; collapsible `kubectl delete` commands |

### Apply Checklist (migrate mode)

Numbered steps produced in order:

1. Fix errors (if any)
2. Update workload env vars (if mappings found)
3. Fix deprecated annotations (if found)
4. Upgrade Global CP to target version
5. Upgrade Zone CPs (max two minor versions per step — e.g. 2.7 → 2.9 → 2.11)
6. Install Gateway API CRDs (if gateway migrations present; experimental channel if `TCPRoute` present)
7. Apply Global CP policies (`resiliency/`, `routing/`, `zero-trust/`, `observability/`)
8. Apply Gateway API resources to every Zone cluster from `all-zones/` (only if present)
9. Apply Zone-origin policies
10. Apply `Mesh` CRs last (enables `spec.meshServices.mode: Exclusive`)
11. Verify traffic health
12. Delete original policy files
13. Plan next upgrade (if not yet at target version)

### Console output labels

| Label | Meaning |
|---|---|
| `[MIGRATED LEGACY]` | Scenario A — legacy sources/destinations |
| `[MIGRATED SUBSET]` | Scenario B — service-identity tag rewrite |
| `[MIGRATED RULES]` | Scenario D — `from[]` → `rules[]` |
| `[MIGRATED MESH]` | Mesh CRD observability extraction |
| `[MIGRATED ES]` | `ExternalService` → `MeshExternalService` |
| `[MIGRATED GW]` | Gateway scenarios (MeshGateway, MeshHTTPRoute, etc.) |
| `[MIGRATED OPA]` | `OPAPolicy` → `MeshOPA` |
| `[ALREADY MIGRATED]` | Scenario C — passed through unchanged |
| `[SKIP]` | No recognised Kuma policies or empty file |
| `[ERROR]` / `[PARTIAL ERROR]` | Transformation failed (all or some documents) |

Per-document inline markers: `[ERROR]`, `[WARN]`, `[WORKLOAD]`, `[ANNOTATION]`

### Output file naming

`<Kind>[-<namespace>]-<name>.yaml` — `/`, `:`, and spaces replaced with `-`.

## 11. Config File

**Resolution order**:
1. `$KUMA_MIGRATOR_CONFIG` environment variable (if set)
2. `~/.config/kuma-migrator.yaml`

**Format**:
```yaml
skip:
  - SomeKind
  - AnotherKind
```

If the file is absent or `skip` is empty, the [default skip list](#default-skip-list) applies. Extra kinds in the config file are merged with the defaults.

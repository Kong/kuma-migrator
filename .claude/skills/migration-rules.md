# Kuma Migrator — Migration Rules

Authoritative reference for transformation logic. Consult before writing or modifying
any scenario transform, detection logic, or deprecation scanner.

---

## 1. Scenario Overview

| Scenario | Constant | Trigger | Output |
|---|---|---|---|
| Legacy | `ScenarioLegacy` | Top-level `sources`/`destinations` or non-`Mesh*` `type` | `targetRef`/`to`/`from` style |
| Subset | `ScenarioSubset` | `Mesh*` kind + `MeshSubset` with service-identity tags | `Dataplane`/`MeshService` targetRefs |
| Passthrough | `ScenarioPassthrough` | `Mesh*` kind + no service-identity tags + no `from[]` | pass through unchanged |
| Rules | `ScenarioRules` | `Mesh*` kind + `from[]` + no service-identity tags | `from[]` → `rules[]` |
| Mesh | `ScenarioMesh` | `kind: Mesh` + embedded observability/passthrough OR missing `meshServices.mode: Exclusive` | standalone companion CRDs + cleaned Mesh |
| ExternalService | `ScenarioExternalService` | `kind: ExternalService` | `kind: MeshExternalService` |
| Gateway | `ScenarioGateway` | `kind: MeshGateway` | `kind: Gateway` (gateway.networking.k8s.io) |
| GatewayInstance | `ScenarioGatewayInstance` | `kind: MeshGatewayInstance` | `GatewayClass` + `MeshGatewayConfig` |
| HTTPRoute | `ScenarioHTTPRoute` | `kind: MeshHTTPRoute` | `kind: HTTPRoute` |
| TCPRoute | `ScenarioTCPRoute` | `kind: MeshTCPRoute` | `kind: TCPRoute` |
| GatewayRoute | `ScenarioGatewayRoute` | `kind: MeshGatewayRoute` | `HTTPRoute`/`TCPRoute` |
| OPAPolicy | `ScenarioOPAPolicy` | `kind: OPAPolicy` (Kong Mesh) | `kind: MeshOPA` |

## 2. Detection Order

`DetectScenario` in `detect.go` must evaluate in this order (first match wins):

1. **Passthrough** — `kind` is a Gateway API native kind (`Gateway`, `HTTPRoute`, `TCPRoute`, etc.) → `ScenarioPassthrough`
2. **ExternalService** — `kind: ExternalService` → `ScenarioExternalService`
3. **Mesh** — `kind: Mesh` + `meshNeedsMigration()` returns true → `ScenarioMesh`; otherwise `ScenarioPassthrough`
4. **Gateway/Instance/Routes** — `kind: MeshGateway/MeshGatewayInstance/MeshHTTPRoute/MeshTCPRoute/MeshGatewayRoute`
5. **OPAPolicy** — `kind: OPAPolicy` → `ScenarioOPAPolicy`
6. **Legacy** — top-level `sources` or `destinations` key present, OR `type` is a non-`Mesh*` policy name → `ScenarioLegacy`
7. **Mesh*-style policies** — has `apiVersion: kuma.io/v1alpha1` + `Mesh*` kind:
   - If any `targetRef` has service-identity tags → `ScenarioSubset`
   - Else if `from[]` is present → `ScenarioRules`
   - Else → `ScenarioPassthrough`

**Service-identity tags** (presence of either triggers Subset detection):
- `kuma.io/service`
- `k8s.kuma.io/service-name`

## 3. ScenarioLegacy — Tag Value → MeshService Name Conversion

### 3.1 `kuma.io/service` tag (Kubernetes format)

Auto-generated by Kuma CP as `<service>_<namespace>_svc_<port>`:

```
kuma.io/service: backend_demo_svc_3001
  → name:      backend
  → namespace: demo
  → port:      3001
```

**Parsing algorithm:**
1. Split on `_svc_` → left part is `<name>_<namespace>`, right part is port
2. Split left part on the **last** `_` to separate namespace (names may contain `_`)
3. `'*'` → `kind: Mesh` (wildcard, not a service reference)

**Universal mode:** `kuma.io/service` values are free-form (not `_ns_svc_port` encoded).
Detect Universal mode by the absence of `k8s.kuma.io/*` labels. Use the raw tag value
as the `MeshService` name.

### 3.2 `k8s.kuma.io/service-name` tag

Directly holds the Kubernetes Service name:

```
tags:
  k8s.kuma.io/service-name: redis
  k8s.kuma.io/namespace: demo   # optional

→ kind: MeshService, name: redis, namespace: demo
```

### 3.3 Non-service tags → `labels`

`MeshSubset` entries with **both** service-identity and other tags: output uses
`kind: MeshService` with `labels` for the non-service refinements:

```yaml
# Input
kind: MeshSubset
tags:
  kuma.io/service: backend_demo_svc_3001
  version: v2

# Output
kind: MeshService
labels:
  kuma.io/display-name: backend
  k8s.kuma.io/namespace: demo
  version: v2
```

If **only** non-service tags: keep `MeshSubset` unchanged (workload selector, not service selector).

## 4. targetRef Mapping Tables

### 4.1 `spec.targetRef` (top-level — "who this policy applies to")

In Kuma 2.13.x, `spec.targetRef` only supports `kind: Mesh` and `kind: Dataplane`.
`MeshService` is **not** valid at the top level.

| Input | Output |
|---|---|
| `kind: Mesh` | `kind: Mesh` (unchanged) |
| `kind: MeshSubset, tags: {kuma.io/service: '*'}` | `kind: Mesh` |
| `kind: MeshSubset, tags: {kuma.io/service: svc_ns_svc_port}` | `kind: Dataplane, name: svc, namespace: ns` |
| `kind: MeshSubset, tags: {k8s.kuma.io/service-name: svc, k8s.kuma.io/namespace: ns}` | `kind: Dataplane, name: svc, namespace: ns` |
| `kind: MeshSubset, tags: {kuma.io/service: X_ns_svc_p, version: v2}` | `kind: Dataplane, labels: {app: X, k8s.kuma.io/namespace: ns, version: v2}` |
| `kind: MeshSubset, tags: {non-service tags only}` | `kind: Dataplane, labels: {<tags>}` |
| `kind: Dataplane, name/labels: {...}` | unchanged |

From old `sources[]` (ScenarioLegacy):

| Old `sources` match | New `spec.targetRef` |
|---|---|
| `kuma.io/service: '*'` | `kind: Mesh` |
| `kuma.io/service: svc_ns_svc_port` | `kind: Dataplane, name: svc, namespace: ns` |
| service tag + other tags | `kind: Dataplane, labels: {app: svc, k8s.kuma.io/namespace: ns, <other tags>}` |

### 4.2 `to[].targetRef` (outbound — "where traffic goes")

| Input | Output |
|---|---|
| `kind: Mesh` | `kind: Mesh` (unchanged) |
| `kind: MeshSubset, tags: {kuma.io/service: svc_ns_svc_port}` | `kind: MeshService, name: svc, namespace: ns` ¹ |
| `kind: MeshSubset, tags: {kuma.io/service: ..., kuma.io/zone: east}` | `kind: MeshService, labels: {kuma.io/display-name: svc, k8s.kuma.io/namespace: ns, kuma.io/zone: east}` |
| Old `destinations: [{match: {kuma.io/service: svc_ns_svc_port}}]` | `kind: MeshService, name: svc, namespace: ns` ¹ |
| Old `destinations: [{match: {kuma.io/service: '*'}}]` | `kind: Mesh` |

¹ Use `name/namespace` when the policy namespace matches the target's namespace.
  Use `labels` (`kuma.io/display-name` + `k8s.kuma.io/namespace`) when the policy is in
  a system namespace or on Global CP (name/namespace would be ambiguous across zones).

### 4.3 `from[].targetRef` (inbound — "who sends traffic")

Same rules as `to[].targetRef`, including the `name/namespace` vs `labels` scoping rule.

### 4.4 `name/namespace` vs `labels` scoping rule

| Policy location | Use |
|---|---|
| Same namespace as target resource | `name` + `namespace` |
| System namespace (`kuma-system`, `kong-mesh-system`) | `labels` |
| Global CP | `labels` (a MeshService with the same name can exist in multiple zones) |

Use `kuma.io/display-name` (not `k8s.kuma.io/service-name`) for cross-namespace label
selection. The CP sets `kuma.io/display-name` on every auto-generated MeshService.

## 5. Policy Type Renaming (ScenarioLegacy)

| Old `type` | New `kind` |
|---|---|
| `Timeout` | `MeshTimeout` |
| `CircuitBreaker` | `MeshCircuitBreaker` |
| `Retry` | `MeshRetry` |
| `TrafficPermission` | `MeshTrafficPermission` |
| `FaultInjection` | `MeshFaultInjection` |
| `RateLimit` | `MeshRateLimit` |
| `HealthCheck` | `MeshHealthCheck` |
| `TrafficLog` | `MeshAccessLog` |
| `TrafficTrace` | `MeshTrace` |
| `TrafficRoute` | **error** — requires manual migration to `MeshHTTPRoute`/`MeshTCPRoute` |
| `ProxyTemplate` | `MeshProxyPatch` |

## 6. TrafficPermission Inversion (ScenarioLegacy only)

`TrafficPermission` has **inverted semantics** vs all other old-style policies:

| Field | Meaning in all other policies | Meaning in TrafficPermission |
|---|---|---|
| `sources` | who the policy applies to → `spec.targetRef` | who is allowed in → `from[].targetRef` |
| `destinations` | where traffic goes → `to[].targetRef` | the service being protected → `spec.targetRef` |

## 7. Splitting for Multiple Sources (ScenarioLegacy)

When `sources[]` has multiple specific (non-wildcard) services, emit **one policy per
source** because `spec.targetRef` accepts only a single reference:

```yaml
# Input: one policy, two sources
sources:
  - match: {kuma.io/service: frontend_demo_svc_8080}
  - match: {kuma.io/service: backend_demo_svc_3001}
destinations:
  - match: {kuma.io/service: redis_demo_svc_6379}

# Output: two policies
---
spec:
  targetRef: {kind: Dataplane, name: frontend, namespace: demo}
  to: [{targetRef: {kind: MeshService, name: redis, namespace: demo}, default: <conf>}]
---
spec:
  targetRef: {kind: Dataplane, name: backend, namespace: demo}
  to: [{targetRef: {kind: MeshService, name: redis, namespace: demo}, default: <conf>}]
```

Name split policies `<original>-0`, `<original>-1`, etc.

## 8. ScenarioRules — `from[]` → `rules[]` (Kuma 2.10+)

### 8.1 Supported kinds (ScenarioRules applies)

The `rules:` block replaced `from:` for inbound traffic in Kuma 2.10 for these five kinds:

| Kind | Notes |
|---|---|
| `MeshAccessLog` | |
| `MeshCircuitBreaker` | |
| `MeshRateLimit` | |
| `MeshTimeout` | |
| `MeshTls` | |

These are the **only** kinds in `rulesAPIMigrationKinds`. Any new kind that gains `rules:` support must be added to that map explicitly.

### 8.2 Excluded kinds — `from:` stays permanent

These two kinds explicitly **do not** support the Rules API and must **never** be added to `rulesAPIMigrationKinds`. Their `from:` block is the intended, permanent API — not a deprecation:

| Kind | Reason |
|---|---|
| `MeshTrafficPermission` | Cannot match subsets of incoming traffic; `from:` is the only inbound mechanism |
| `MeshFaultInjection` | Same — inbound-only policy; no Rules API equivalent |

When these kinds have a `from:` block they fall through to `ScenarioPassthrough` (pass through unchanged).

### 8.3 Transformation

```yaml
# Input (from[]-based, deprecated in 2.10)
spec:
  targetRef:
    kind: Dataplane
    labels:
      app: backend
  from:
    - targetRef:
        kind: Mesh
      default:
        connectTimeout: 5s

# Output (rules-based)
spec:
  targetRef:
    kind: Dataplane
    labels:
      app: backend
  rules:
    - default:
        connectTimeout: 5s
```

**Warning:** If `from[]` has multiple entries with different `targetRef` values (source-specific
rules), each entry collapses into a `rules[]` entry without source discrimination. Emit a
warning: the source-specific targeting is lost and all inbound traffic will be affected.

`to[]` is preserved unchanged alongside `rules[]`.

## 9. ScenarioMesh — Mesh CRD Transformation

`meshNeedsMigration()` returns `true` when any of these conditions hold:
- `spec.metrics` is set
- `spec.tracing` is set
- `spec.logging` is set
- `spec.networking.outbound.passthrough` is set
- `spec.meshServices.mode` is not `"Exclusive"`

**Extracted companion CRDs:**

| Mesh section | Output kind |
|---|---|
| `spec.metrics` (prometheus) | `MeshMetric` |
| `spec.tracing` (zipkin/datadog) | `MeshTrace` |
| `spec.logging` (file/tcp) | `MeshAccessLog` |
| `spec.networking.outbound.passthrough: false` | `MeshPassthrough` with `passthroughMode: None` |

**Always injected:**
- `spec.meshServices.mode: Exclusive` is set on every processed Mesh CRD (with a warning
  to verify all policies and workloads are migrated first).

Companion CRD naming: `<meshName>-metrics`, `<meshName>-trace`, `<meshName>-accesslog`,
`<meshName>-passthrough`. Validate names against RFC 1035/1123 and emit a warning if
the mesh name would produce an invalid name.

## 10. ScenarioExternalService → MeshExternalService

Structural mapping:

| ExternalService field | MeshExternalService field |
|---|---|
| `networking.address` (host:port) | `spec.match.port.value` + `spec.match.hostname` |
| `networking.tls` | `spec.tls` |

**Guard:** If `networking.address` starts with `unix://`, emit an error — Unix socket
addresses are not supported in MeshExternalService.

## 11. ScenarioOPAPolicy → MeshOPA (Kong Mesh)

```yaml
# Input
spec:
  conf:
    policies:
      - inlineString: "<rego>"
    agentConfig:
      inlineString: "<config>"   # optional

# Output
spec:
  default:
    appendPolicies:
      - rego:
          inlineString: "<rego>"
    agentConfig:
      inlineString: "<config>"   # if present
```

`targetRef` is preserved unchanged. `MeshOPA` produces static config; dynamic config
via `MeshOPAPolicy` requires manual setup.

## 12. ScanForDeprecations (post-pass on every output document)

`ScanForDeprecations` in `deprecation.go` is called on **every output document** regardless
of scenario. It emits warnings (never transforms) for:

| Kind | Field / condition | Version | Action |
|---|---|---|---|
| `MeshMetric` | `spec.default.sidecar.regex` | 2.7 | **auto-fixed** → `sidecar.profiles.exclude[{type: Regex, match: <regex>}]` |
| `MeshHealthCheck` | `spec.default.healthyPanicThreshold` | 2.10 | warn → move to `MeshCircuitBreaker` |
| `MeshTrust` | `spec.origin` non-empty | 2.13 | warn → deprecated, moved to `status.origin` |
| `MeshTrafficPermission`, `MeshFaultInjection` | `from[].targetRef.kind: MeshService` | 2.7 | warn → deprecated |
| `MeshTrafficPermission` | `action: ALLOW / DENY / ALLOW_WITH_SHADOW_DENY` (uppercase) | KM 2.1 | warn → use `Allow`/`Deny` |
| `MeshLoadBalancingStrategy` | `hashPolicies[].type: SourceIP` | 2.10 | warn → use `Connection` |
| `Dataplane` | `transparentProxying.redirectPortInboundV6` | 2.9 | warn → field removed |
| Any resource | `metadata.annotations` with `kuma.io/*: "yes"` or `"no"` | 2.9 | warn → use `"true"`/`"false"` |
| Any `Mesh*` kind | `spec.targetRef.kind: MeshSubset` without service-identity tags | 2.10 | warn → use `Dataplane` with labels |

## 13. Producer/Consumer Policy Roles

The form used in `to[].targetRef` determines the Kuma-assigned policy role:

| Targeting form | Policy namespace | Role |
|---|---|---|
| `name: svc, namespace: ns` (same ns as MeshService) | app namespace | `producer` — synced to all zones |
| `labels: {kuma.io/display-name: svc, ...}` | any | `consumer` — local override |
| policy in system namespace | `kuma-system` / `kong-mesh-system` | `system` |
| no `to[]`/`from[]` | any | `workload-owner` |

## 14. `meshServices.mode` on the `Mesh` resource

| Mode | Behaviour |
|---|---|
| `ReachableBackends` *(default)* | Both `kuma.io/service` tags and `MeshService` resources active. Policies can match either. |
| `Exclusive` | **Only** `MeshService`/`MeshExternalService`/`MeshMultizoneService`. Legacy tag matching disabled. |

`kuma-migrator` always sets `spec.meshServices.mode: Exclusive` on every Mesh CRD it
processes. This is required for MeshService-based policies to take effect.

**Before enabling Exclusive mode**, confirm all policies and workload env-var addresses
in the mesh have been migrated.

## 15. RFC 1035 / RFC 1123 Name Validation (Kuma 2.10+)

**RFC 1035 kinds** (stricter — alphabetic start, max 63 chars):
`Mesh`, `MeshService`, `MeshExternalService`, `MeshMultizoneService`

**RFC 1123 kinds** (all others — alphanumeric start allowed):
Everything else

`ValidateResourceName(name, kind)` in `validate.go` returns a warning string if the
name violates the applicable standard. Call it:
1. When reading input resource names in all `TransformXxx` functions
2. When generating output resource names (e.g. `<meshName>-metrics`)

## 16. Address Mapping Scanner

Scans `Deployment`, `StatefulSet`, `DaemonJob` env var values for patterns matching
`<svc>_<ns>_svc_<port>.mesh` (legacy kuma.io/service encoded addresses).

Produces an `AddressMapping` entry with:
- `OldToken` — the encoded service token (e.g. `backend_demo_svc_3001`)
- `ExplicitPort` — the `.mesh` suffix and any explicit port
- `NewK8sAddress()` — `backend.demo.svc.cluster.local:3001`
- `NewMeshAddress()` — `backend.demo.svc.mesh.<zone>`

Deduplicated by `OldToken` in the final report.
